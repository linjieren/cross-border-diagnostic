# B-003 反馈去重管道集成

## 用户故事

作为产品负责人，我希望系统能自动检测并标记重复反馈，并在管理页面展示重复关系，以便我不需要手动判断哪些反馈是同一条问题。

## Acceptance Criteria

- [ ] **数据库变更**：`feedbacks` 表新增字段
  - `duplicate_of` (String, nullable, FK → feedbacks.id)：指向该重复组的主反馈 ID（反馈 ID 格式为 `fbk-{timestamp}-{hash}`）
  - `duplicate_group_id` (String, nullable)：同一重复组的共享标识（如 hash 或 uuid），便于批量查询
- [ ] **后端 API**：`POST /api/feedback/dedup` —— 对指定反馈运行去重检测
  - 输入：`{ feedbackId: string }`
  - 输出：`{ duplicateOf: string | null, duplicateGroupId: string | null, confidence: number, reason: string }`
  - 去重逻辑复用反馈线分类器规则：文本相似度 >= 80%（Jaccard 或 Levenshtein，与 classifier.md 对齐）或截图 perceptual hash 差值 < 5
  - 若找到重复，更新该反馈的 `duplicate_of` 和 `duplicate_group_id`
  - 若找不到，字段保持 null
- [ ] **后端 API**：`GET /api/feedback` 列表接口扩展
  - 响应中每条反馈增加 `duplicateOf` 和 `duplicateGroupId` 字段
  - 新增查询参数 `?dedupStatus=all|unique|duplicates`，支持筛选"全部/仅主反馈/仅重复项"
- [ ] **前端管理页面**：`/admin/feedback` 列表页扩展
  - 重复反馈行以视觉标识（如灰色文字、折叠按钮或 "与 #X 重复" 标签）
  - 支持按 "去重状态" 筛选（全部 / 仅主反馈 / 仅重复项）
  - 重复项可折叠到主反馈下，减少列表噪音
- [ ] **前端管理页面**：反馈详情页扩展
  - 若当前反馈是主反馈，展示"重复组"面板，列出所有指向它的重复反馈
  - 若当前反馈是重复项，展示"主反馈"链接，一键跳转
- [ ] **批量去重巡检**：`POST /api/feedback/dedup-all`（或集成到现有反馈评估 pipeline）
  - 对未标记去重状态的存量反馈批量运行检测
  - 返回处理的条数和发现的重复组数
  - 受权限控制（仅管理员）

## 技术边界

- 去重算法由反馈线分类器定义，开发线只负责集成调用或复用其规则
- 不删除重复反馈，只标记关系，保留原始数据
- `duplicate_of` 指向的必须是已存在的 feedback ID
- 去重检测是异步可接受的（非实时），允许在反馈提交后延迟几秒到几分钟完成
