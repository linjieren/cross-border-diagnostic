# B-002 反馈管理前端页面

## 用户故事
作为产品负责人和反馈运营人员，我希望有一个管理页面能查看、筛选、评估已提交的用户反馈，以便高效协作完成反馈评估和优先级判断。

## Acceptance Criteria

### 反馈列表页（主页面）
- [ ] 路由 `/admin/feedback`，不依赖 `VITE_FEATURE_FEEDBACK_BUTTON` 环境变量
- [ ] 分页展示反馈列表，每页默认 20 条
- [ ] 每条显示：id（缩略）、标题、类型、严重度、状态、提交时间
- [ ] 支持按 status 筛选：new / evaluated / accepted / rejected / in_development / verified / closed
- [ ] 支持按 type 筛选：bug / feature_request / feedback / question
- [ ] 支持按 severity 筛选：critical / high / medium / low
- [ ] 支持关键词搜索（标题模糊匹配）
- [ ] 列表为空时展示空状态提示

### 反馈详情页
- [ ] 点击列表条目进入详情页，路由 `/admin/feedback/:id`
- [ ] 展示完整反馈信息：标题、正文、类型、严重度、状态、截图（如有）、提交者信息、user_agent、提交时间
- [ ] 截图支持点击放大查看

### 评估操作
- [ ] 产品负责人可修改 feedback 的 status（下拉选择）
- [ ] 产品负责人可修改 feedback 的 category/type
- [ ] 产品负责人可修改 feedback 的 priority
- [ ] 产品负责人可添加评估备注（notes 字段）
- [ ] 修改后通过 API 持久化保存

### 技术约束
- [ ] 对接已有 GET /api/feedback（分页列表）和 GET /api/feedback/{id}（详情）端点
- [ ] 评估更新需后端提供 PATCH /api/feedback/{id} 端点（如不存在请新增）
- [ ] 前端使用项目已有技术栈：Vite + React 18 + TypeScript

## 优先级
p1_high — 反馈线评估引擎就绪后，产品+反馈线需要管理页面才能高效协作

## 关联反馈 ID
product_planned

## 预估工作量
M（1-2 天：后端 PATCH 端点 + 前端列表页 + 详情页 + 筛选逻辑）

## 状态
ready

## 备注
- 后端数据模型已有 status/category/priority/notes 字段（Prisma schema），前端管理页面直接对接
- 列表页参考常见 admin 面板风格即可，不需要设计稿
- 这是一个内部管理页面，不需要 FeedbackButton 那样的 feature gate
