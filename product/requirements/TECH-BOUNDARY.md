## 技术约束

### iframe 加载限制
- 大量网站设置了 `X-Frame-Options: DENY` 或 `Content-Security-Policy: frame-ancestors`，阻止被 iframe 嵌入
- **解决方案**：
  - 方案 A：通过代理服务器加载目标网站（重写响应头）
  - 方案 B：当检测到 iframe 被阻止时，提示用户"该网站不支持 iframe 预览，请手动在浏览器中打开对应页面查看"
  - 方案 C：使用无头浏览器截图替代实时 iframe

### 网站内容抓取限制
- 目标网站可能通过 robots.txt 限制爬虫
- SPA（React/Vue/Angular）网站需要等待 JS 执行后才能获取完整 DOM
- **解决方案**：使用无头浏览器（Puppeteer/Playwright）抓取，而非简单 HTTP 请求

### CORS 限制
- 从前端直接请求第三方网站 API 会触发 CORS
- **解决方案**：所有对外部网站的检测请求必须通过后端代理

### 代码部署指导的准确性
- 不同建站平台（WordPress、Shopify、自建站）的代码插入位置不同
- **解决方案**：生成部署代码时附带主流平台（WordPress、Shopify、Wix、自建 HTML）的分别说明

## 数据安全与合规

- 平台会抓取并分析客户网站内容，需明确告知用户数据使用范围
- 诊断报告可能包含客户网站的截图和内容摘要，需确保存储安全
- 建议不存储客户网站的完整源码，仅保留分析结果和截图

## 性能预估

| 场景 | 预估响应时间 | 说明 |
|------|-------------|------|
| 单次延迟检测 | 3-10 秒 | 取决于目标市场距离和网站复杂度 |
| 单页面模块分析 | 1-3 秒 | DOM 解析 + 关键词匹配 |
| 完整报告生成（PDF） | 5-15 秒 | 取决于报告页数，建议异步生成 |
| 全站 4 模块完整诊断 | 15-45 秒 | 用户可逐页浏览，非一次性等待 |

## 建议技术栈（供开发线参考）

- **前端**：React 18 + TypeScript（左右分屏布局、iframe 管理、实时状态同步）
- **后端**：Node.js / Python（API 代理、网站抓取、报告生成）
- **报告生成**：Puppeteer / Playwright（HTML → PDF）+ markdown-it（HTML → Markdown）
- **抓取引擎**：Playwright（支持 SPA、自动等待、截图）
- **数据库**：PostgreSQL（客户诊断记录缓存、报告模板存储）
- **部署**：Docker Compose（快速启动）
