# 需求条目模板

每个需求条目必须包含以下字段。开发线拿到需求后不需要再猜"做成什么样"。

---

## 条目格式

### 标题
<!-- 一句话描述，动词开头，用户视角：如"用户可以导出反馈列表为 CSV" -->

### 用户故事
<!-- 格式：作为 <角色>，我希望 <功能>，以便 <价值> -->

### Acceptance Criteria
<!-- 可验证的、无歧义的验收条件。每条一个 checkbox -->

- [ ] 条件 1
- [ ] 条件 2

### 优先级
<!-- p0_critical / p1_high / p2_medium / p3_low -->

| 优先级 | 判断标准 |
|--------|----------|
| p0_critical | 核心功能不可用、数据丢失、安全漏洞 |
| p1_high | 严重影响体验，大部分用户会遇到 |
| p2_medium | 部分用户体验下降，有 workaround |
| p3_low | 小问题、锦上添花的建议 |

### 关联反馈 ID
<!-- 如果来自用户反馈，写反馈 ID；如果是产品规划，写 product_planned -->

### 预估工作量
<!-- S(半天) / M(1-2天) / L(3-5天) / XL(>5天) -->

### 状态
<!-- draft → ready → in_development → verified → closed -->

### 备注
<!-- 技术约束、设计参考、边界条件等 -->
