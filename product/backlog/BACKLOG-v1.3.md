## v1.3 Backlog（按优先级排序）

### p1_high · 立即推进（1-2 个迭代）

#### FEAT-301 诊断历史记录页

- **用户故事**：作为已登录用户，我希望查看我之前做过的所有诊断记录，并能重新打开或下载历史报告，以便追踪网站改进进度。
- **Acceptance Criteria**：
  - 个人中心新增「我的诊断」页面
  - 列表展示：网站 URL、目标市场、诊断时间、4 模块完成状态、报告下载入口
  - 支持按时间倒序排列、按目标市场筛选
  - 点击单条记录可进入「诊断回放」—— 复现当时的左右分屏结果（不重新爬取，直接加载缓存结果）
  - 支持删除单条记录（软删除，保留 30 天可恢复）
- **优先级**：p1_high
- **预估工作量**：M（2-3 天）
- **依赖**：v1.2 用户系统、v1.2 诊断结果持久化存储
- **外部依赖**：无
- **状态**：draft

---

#### FEAT-302 微信登录完整接入

- **用户故事**：作为中国大陆用户，我希望通过微信扫码直接登录，无需记忆额外密码，以便快速使用平台。
- **Acceptance Criteria**：
  - 登录/注册弹窗增加「微信扫码登录」入口
  - 扫码后自动完成：获取 openid → 查找或创建用户 → 发放 JWT Token → 登录态写入 Cookie
  - 已存在邮箱账号的用户：支持「绑定微信」功能（个人中心 → 账号安全）
  - 未关注公众号的用户：首次登录后引导关注（可选，取决于运营策略）
  - 微信登录态过期处理：静默刷新或引导重新扫码
- **优先级**：p1_high
- **预估工作量**：M（2-3 天）
- **依赖**：v1.2 用户系统（已有 JWT + 邮箱登录基础）
- **外部依赖**：微信开放平台 appid / appsecret（需用户提供）
- **状态**：draft
- **备注**：
  - 需用户提供「微信开放平台」或「微信公众平台服务号」的 appid + appsecret
  - 若用户暂无，可先接入微信 JS-SDK 的扫码能力，或降级为「微信内网页授权」（snsapi_userinfo）

---

#### FEAT-303 多语言支持（英文优先）

- **用户故事**：作为非中文母语用户（海外 SME），我希望平台界面、诊断报告和优化建议都支持英文，以便我的团队直接使用。
- **Acceptance Criteria**：
  - 前端界面全面 i18n：所有硬编码中文文本提取到翻译文件
  - 支持语言切换：中文（默认）↔ English，切换后持久化到用户偏好
  - 诊断报告双语生成：PDF / Markdown / HTML 报告根据用户语言偏好输出对应语言
  - 模块内优化建议和模板文案（如行业表单模板、部署代码注释）支持英文版
  - URL 不携带语言参数，通过用户偏好或浏览器 `Accept-Language` 自动判断
- **优先级**：p1_high
- **预估工作量**：L（3-5 天）—— 涉及前端全量文本提取 + 后端报告模板双语化
- **依赖**：v1.2 全量前端页面 + 报告引擎
- **外部依赖**：无（可自建翻译，不依赖第三方翻译 API）
- **状态**：draft
- **备注**：
  - 日文、西班牙文等二期再扩展
  - 报告中的 YouTube 教程链接（MOD-001）保持英文原版即可

### p2_medium · 中期推进（2-3 个迭代后）

#### FEAT-304 真实邮件发送通道

- **用户故事**：作为平台用户，我希望诊断完成后自动收到一封邮件通知，包含报告摘要和下载链接，以便我与团队分享。
- **Acceptance Criteria**：
  - 诊断完成后，自动发送邮件到用户注册邮箱
  - 邮件内容：诊断概要（4 模块评分摘要）+ 报告 PDF 下载链接 + CTA「查看完整诊断」
  - 支持邮件模板：诊断完成通知、欢迎注册、密码重置、周报/月报推送（可选）
  - 发送失败时重试 3 次，仍失败则记录日志并提示用户「邮件发送失败，请检查邮箱」
- **优先级**：p2_medium
- **预估工作量**：S-M（1-2 天）
- **依赖**：v1.2 用户系统、FEAT-301（诊断历史）
- **外部依赖**：邮件服务商 API Key（SendGrid / AWS SES / 阿里云邮件推送）
- **状态**：draft
- **备注**：
  - 推荐 SendGrid（海外用户为主，免费额度 100 封/天）
  - 短信通道优先级低于邮件（海外 SME 主要用邮件），v1.3 不强制包含短信

#### FEAT-305 管理员后台

- **用户故事**：作为平台运营人员，我希望有一个管理后台查看所有用户的诊断记录、活跃度和系统健康状态，以便运营和排障。
- **Acceptance Criteria**：
  - 独立路由 `/admin/dashboard`（已有 admin 权限校验）
  - 用户管理页：注册用户列表、注册来源（邮箱/微信）、最近登录时间
  - 诊断记录总览：总诊断次数、按模块统计完成率、按目标市场分布
  - 系统健康：最近 7 天 API 错误率、平均诊断耗时、iframe 加载失败率
  - 支持导出用户列表和诊断记录为 CSV
- **优先级**：p2_medium
- **预估工作量**：M（2-3 天）
- **依赖**：v1.2 用户系统 + 诊断数据持久化
- **外部依赖**：无
- **状态**：draft

### p3_low / 需用户决策

#### FEAT-306 支付/订阅系统

- **用户故事**：作为平台运营方，我希望对高级功能（如批量诊断、API 调用、品牌定制报告）收费，以便平台可持续运营。
- **Acceptance Criteria**：
  - 支持至少两种付费模式：按次付费（单次深度诊断）和订阅制（月度/年度无限诊断）
  - 集成第三方支付：Stripe（海外信用卡）+ 微信支付（国内用户）
  - 订阅管理：升级/降级/取消、到期提醒、发票下载
  - 免费增值模式：基础诊断免费，高级功能（如整改报告导出、API 访问）需付费
  - 管理员后台新增「订单管理」和「收入统计」
- **优先级**：p3_low（需商业模式确认）
- **预估工作量**：L（4-6 天）
- **依赖**：FEAT-305（管理员后台）
- **外部依赖**：Stripe / 微信支付商户号、商户 API Key
- **状态**：draft
- **决策标记**：**需用户决策**
  - 问题 1：付费模式偏好——按次付费 vs 订阅制 vs 免费增值？
  - 问题 2：免费版功能边界——哪些诊断功能免费，哪些收费？
  - 问题 3：目标客单价区间？

## 其他候选方向（p3 / 未来考虑）

| 方向 | 说明 | 暂不推进原因 |
|------|------|-------------|
| 诊断报告分享链接 | 生成公开/私密的报告分享 URL，增强传播 | 有泄露客户网站信息的风险，需先确认隐私策略 |
| 批量诊断 / API 开放 | 一次提交多个 URL 或调用 API 批量诊断 | 依赖 FEAT-306 付费体系，否则成本不可控 |
| 定时重新诊断 | 每周/每月自动重新诊断同一网站并生成对比报告 | 需要 Cron + 邮件通道就绪 |
| 行业模板库扩展 | 增加更多行业（医疗、教育、物流）的表单/内容模板 | 内容生产成本高，先验证现有 4 个行业使用率 |
| 团队协作 / 子账号 | 企业版支持多成员共享诊断记录 | 依赖 FEAT-306 订阅体系 |
