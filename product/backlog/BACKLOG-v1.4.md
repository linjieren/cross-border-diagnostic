## FEAT-301 诊断历史记录页

### 用户故事

作为已登录用户，我希望查看我之前做过的所有诊断记录，并能重新打开或下载历史报告，以便追踪网站改进进度。

### 使用场景

| 场景 | 用户行为 | 预期结果 |
|------|---------|---------|
| 场景 A：回顾上周诊断 | 用户登录后进入「我的诊断」，看到按时间倒序排列的诊断列表 | 可快速定位到上周的诊断记录 |
| 场景 B：下载历史报告 | 用户在列表中找到某条记录，点击「下载报告」 | 直接下载当时的 PDF / Markdown 报告，无需重新诊断 |
| 场景 C：诊断回放 | 用户点击单条记录，进入诊断回放页 | 复现当时的左右分屏界面，右侧显示当时的诊断结果 |
| 场景 D：清理旧记录 | 用户删除一条不再需要的诊断记录 | 记录从列表中消失，但 30 天内可在「回收站」恢复 |

### 功能边界

**包含**：
- 诊断历史列表页（个人中心 /history）
- 诊断回放页（复现左右分屏结果）
- 报告下载（复用 v1.2 报告引擎，不重新生成）
- 删除与恢复（软删除）
- 列表筛选与排序

**不包含**：
- 定时自动重新诊断（属于未来候选方向）
- 诊断结果对比（两次诊断之间的差异分析）
- 分享诊断给其他人（属于报告分享链接候选方向）

### Acceptance Criteria

#### 1. 诊断历史列表页
- [ ] 页面路由：`/history`（已存在前端页面，需补齐后端 API 和交互）
- [ ] 列表字段：网站 URL（截断显示，hover 显示完整）、目标市场（国旗 + 国家名）、诊断时间（相对时间，如"3 天前"）、4 模块完成状态（4 个小圆点，绿色=完成，灰色=未检测）、操作按钮（查看 / 下载 / 删除）
- [ ] 默认排序：按诊断时间倒序，最新在前
- [ ] 分页：每页 10 条，底部页码导航
- [ ] 空状态：当用户无历史记录时，显示引导文案"你还没有进行过诊断，去首页开始第一次检测吧" + 跳转按钮
- [ ] 筛选器：按目标市场（下拉，展示用户诊断过的所有国家）、按时间范围（近 7 天 / 近 30 天 / 全部）

#### 2. 诊断回放页
- [ ] 点击列表中「查看」进入回放页，路由 `/history/:diagnosisId`
- [ ] 页面布局复用诊断工作台左右分屏结构
- [ ] 左侧 iframe：加载当时诊断的网站 URL（使用当时的缓存 HTML 快照，避免重新爬取）
- [ ] 右侧诊断面板：展示当时保存的诊断结果 JSON，直接渲染（不触发新的检测流程）
- [ ] 顶部显示：网站 URL、目标市场、诊断时间、"历史诊断"标签
- [ ] 提供「重新诊断」按钮：跳转回工作台，使用相同的 URL + 目标市场重新执行完整检测

#### 3. 报告下载
- [ ] 列表中每条记录显示「下载报告」下拉菜单：PDF / Markdown / HTML
- [ ] 点击后直接下载当时生成的报告文件（从存储中读取，不重新生成）
- [ ] 下载文件名格式：`诊断报告_{domain}_{date}.{ext}`，如 `诊断报告_example.com_20260531.pdf`

#### 4. 删除与恢复
- [ ] 每条记录提供「删除」按钮，点击后二次确认弹窗
- [ ] 删除为软删除：数据库中标记 `deletedAt` 字段，前端列表不再展示
- [ ] 页面底部提供「回收站」入口（或筛选器选项），展示 30 天内删除的记录
- [ ] 回收站中支持「恢复」和「彻底删除」
- [ ] 超过 30 天的软删除记录由后台任务自动物理清理

#### 5. 数据持久化要求
- [ ] 诊断完成后，诊断结果 JSON 和生成的报告文件（PDF/Markdown/HTML）必须持久化到存储
- [ ] 每个诊断记录关联：userId、websiteUrl、targetMarket、diagnosisResult（JSON）、reportFiles（存储路径数组）、createdAt、updatedAt、deletedAt
- [ ] 单次诊断的完整数据（含 HTML 快照、截图、结果 JSON）预估存储：5-20MB，需设置单用户存储上限（如 500MB，超限提示清理旧记录）

### 接口契约（供后端参考）

```
GET /api/diagnoses
  Query: { page=1, limit=10, market?, startDate?, endDate?, includeDeleted?=false }
  Response: { items: Diagnosis[], total: number, page, limit }

GET /api/diagnoses/:id
  Response: { id, websiteUrl, targetMarket, result: DiagnosisResult, createdAt, reportFiles: string[] }

DELETE /api/diagnoses/:id
  Body: { permanent?: false }
  Response: { success: true, deletedAt }

POST /api/diagnoses/:id/restore
  Response: { success: true }
```

### 优先级
p1_high（v1.4 首个迭代首位）

### 预估工作量
M（2-3 天）

### 依赖
- v1.2 用户系统（JWT 鉴权）
- v1.2 诊断结果需要持久化（当前可能仅缓存在会话/Redis 中）
- v1.2 报告引擎（报告生成后需写入持久存储）

### 外部依赖
无

## FEAT-302 微信登录完整接入

### 用户故事

作为中国大陆用户，我希望通过微信扫码直接登录，无需记忆额外密码，以便快速使用平台。

### 使用场景

| 场景 | 用户行为 | 预期结果 |
|------|---------|---------|
| 场景 A：新用户微信登录 | 用户点击「微信登录」，扫码授权 | 自动创建账号并登录，跳转诊断工作台 |
| 场景 B：老用户绑定微信 | 已登录的邮箱用户在个人中心点击「绑定微信」 | 扫码后该账号新增微信登录方式 |
| 场景 C：微信用户补全邮箱 | 仅通过微信登录的用户，首次进入个人中心 | 提示「请绑定邮箱，以便接收诊断报告」 |
| 场景 D：微信登录态过期 | 用户 7 天后再次访问，微信 Token 已过期 | 静默刷新成功则继续，失败则引导重新扫码 |
| 场景 E：微信内浏览器打开 | 用户在微信内置浏览器中访问平台 | 自动调用微信网页授权，无需扫码，一键登录 |

### 功能边界

**包含**：
- 微信开放平台扫码登录（PC 端）
- 微信内网页授权登录（移动端微信浏览器）
- 微信绑定/解绑（个人中心）
- 微信登录态管理与过期处理
- 新用户自动注册 + 老用户账号合并

**不包含**：
- 微信小程序登录（未来可考虑）
- 微信支付（属于 FEAT-306）
- 公众号菜单/消息推送（属于运营侧，v1.4 不包含）
- 微信二维码 ticket 的持久化存储（可使用内存缓存，失败时重新生成）

### Acceptance Criteria

#### 1. 登录/注册弹窗
- [ ] 登录弹窗提供两个入口：「邮箱登录」和「微信登录」
- [ ] 「微信登录」入口点击后，弹窗内展示微信二维码（由后端调用微信 API 生成临时二维码）
- [ ] 二维码有效期 5 分钟，过期后显示「二维码已过期，点击刷新」
- [ ] 用户扫码后，前端通过 WebSocket 或轮询（推荐 WebSocket）接收登录成功事件
- [ ] 登录成功后弹窗关闭，页面刷新登录态，用户头像和昵称显示在导航栏

#### 2. 微信登录流程（PC 端扫码）
- [ ] 后端调用微信开放平台「网站应用扫码登录」接口，生成带 scene_id 的二维码
- [ ] 用户扫码 → 微信确认授权 → 微信回调平台后端 with `code`
- [ ] 后端用 `code` 换 `access_token` + `openid`
- [ ] 用 `openid` 查询用户表：
  - 已存在 → 生成 JWT Token，返回用户信息
  - 不存在 → 自动创建用户（nickname=微信昵称，avatar=微信头像，email 留空标记为待补全），生成 JWT Token
- [ ] 通过 WebSocket 推送登录成功消息给前端，前端写入 Cookie/LocalStorage

#### 3. 微信内网页授权（微信内置浏览器）
- [ ] 检测到 `navigator.userAgent` 含 `MicroMessenger` 时，自动走「静默授权」流程
- [ ] 后端引导至 `https://open.weixin.qq.com/connect/oauth2/authorize?scope=snsapi_userinfo`
- [ ] 用户授权后回调，后端获取用户信息并登录/注册
- [ ] 全程无需用户手动扫码，体验无缝

#### 4. 账号绑定与合并
- [ ] 已登录的邮箱用户可在「个人中心 → 账号安全」点击「绑定微信」
- [ ] 扫码授权后，该用户的账号同时支持邮箱登录和微信登录
- [ ] 一个微信 openid 只能绑定一个平台账号（若已绑定其他账号，提示「该微信已绑定其他账号，请先解绑」）
- [ ] 支持「解绑微信」，解绑后该账号恢复仅邮箱登录，但需验证密码或邮箱验证码

#### 5. 邮箱补全引导
- [ ] 仅通过微信注册的用户（email 为空），首次登录后首页顶部显示轻提示：「建议你绑定邮箱，以便接收诊断报告和找回账号」
- [ ] 个人中心中「邮箱」字段显示「未绑定」状态，点击后支持输入邮箱 + 发送验证码 + 绑定
- [ ] 未绑定邮箱的用户，FEAT-304 邮件发送功能对该用户跳过

#### 6. 登录态管理
- [ ] JWT Token 有效期：accessToken 2 小时，refreshToken 7 天
- [ ] 微信 access_token 有效期 2 小时，后端需缓存并自动刷新
- [ ] Token 过期后，前端自动调用 `/api/auth/refresh`，使用 refreshToken 换取新 accessToken
- [ ] 若 refreshToken 也过期，清除登录态，引导重新登录

### 接口契约（供后端参考）

```
POST /api/auth/wechat/qrcode
  Response: { qrcodeUrl: string, sceneId: string, expiresAt: number }

POST /api/auth/wechat/callback
  Body: { code: string }
  Response: { user: User, accessToken: string, refreshToken: string }

POST /api/auth/wechat/bind
  Headers: Authorization: Bearer {accessToken}
  Body: { code: string }
  Response: { success: true, wechatInfo: { nickname, avatar, openid } }

POST /api/auth/wechat/unbind
  Headers: Authorization: Bearer {accessToken}
  Body: { password: string }  // 验证身份
  Response: { success: true }

POST /api/auth/refresh
  Body: { refreshToken: string }
  Response: { accessToken: string, refreshToken: string }
```

### 优先级
p1_high（v1.4 首个迭代第二位，可与 FEAT-301 并行）

### 预估工作量
M（2-3 天）

### 依赖
- v1.2 用户系统（用户表、JWT 体系）
- 需后端支持 WebSocket 或轮询接口用于二维码登录状态推送

### 外部依赖
- **微信开放平台 appid + appsecret**（网站应用类型，需用户提供）
- **已备案域名**（微信回调需备案域名，需用户确认）
- **备注**：若用户暂无微信开放平台资质，v1.4 迭代内可降级为仅支持「微信内浏览器网页授权」（需公众号/服务号 appid，门槛较低），PC 端扫码登录顺延到 v1.5。

## FEAT-303 多语言支持（英文优先）

### 用户故事

作为非中文母语用户（海外 SME），我希望平台界面、诊断报告和优化建议都支持英文，以便我的团队直接使用。

### 使用场景

| 场景 | 用户行为 | 预期结果 |
|------|---------|---------|
| 场景 A：首次访问自动识别 | 英国用户首次访问，浏览器语言为 en-GB | 平台自动展示英文界面 |
| 场景 B：手动切换语言 | 中国用户进入设置页，切换语言为 English | 全站立即切换为英文，偏好保存到账号 |
| 场景 C：英文诊断报告 | 英文偏好用户完成诊断后下载 PDF 报告 | 报告全文为英文，包括模块名称、评分说明、优化建议 |
| 场景 D：代码部署注释 | 英文偏好用户查看 MOD-002 的一键部署代码 | 代码中的注释和粘贴位置说明为英文 |

### 功能边界

**包含**：
- 前端 UI 文本全量 i18n（zh-CN / en-US）
- 后端诊断报告模板双语化
- 模块内优化建议文案双语化
- 行业模板（表单模板、内容模板）双语化
- 语言偏好持久化

**不包含**：
- 目标网站内容的翻译（我们分析的是用户自己的网站，不需要翻译）
- YouTube 教程标题/描述的翻译（保持英文原版）
- 日文、西班牙文等其他语言（v1.5 扩展）
- 后端日志、错误信息的翻译（保持英文，便于开发排障）

### Acceptance Criteria

#### 1. 语言切换机制
- [ ] 导航栏右侧增加语言切换按钮：🇨🇳 中文 / 🇺🇸 English
- [ ] 未登录用户：语言偏好保存到 `localStorage`，下次访问自动恢复
- [ ] 已登录用户：语言偏好保存到用户表的 `languagePreference` 字段，跨设备同步
- [ ] 首次访问时，根据浏览器 `Accept-Language` 自动判断：
  - `zh-CN`, `zh-TW`, `zh-HK` → 中文
  - `en-*` 或其他 → 英文
  - 可覆盖：用户手动切换后不再自动判断
- [ ] 切换语言后，页面无需刷新，文本即时切换（前端 i18n 框架实现）

#### 2. 前端 UI i18n 范围
- [ ] 所有硬编码中文文本提取到翻译文件，文件结构：
  ```
  src/i18n/
    zh-CN.json    // 中文翻译源
    en-US.json    // 英文翻译源
  ```
- [ ] 必须覆盖的文本范围：
  - 导航栏（首页、工作台、我的诊断、个人中心、登录/注册）
  - 入口页（输入表单标签、placeholder、按钮文案、提示信息）
  - 诊断工作台（模块名称、状态指示器、检测中提示）
  - 诊断结果面板（各模块评分标题、建议文案、CTA 按钮）
  - 报告下载页（格式选择、下载按钮）
  - 个人中心（各项设置标签、按钮）
  - 登录/注册弹窗（所有表单标签、错误提示、按钮）
- [ ] 日期/时间格式：中文用 `YYYY年MM月DD日`，英文用 `May 31, 2026`
- [ ] 数字格式：千分位、百分比等按语言习惯展示

#### 3. 后端报告模板双语化
- [ ] 报告引擎（PDF / Markdown / HTML）支持按 `language` 参数输出对应语言
- [ ] 报告内文本范围：
  - 封面（诊断报告标题、网站、目标市场、诊断时间）
  - 各模块评分卡标题和说明
  - 优化建议文案
  - 部署代码的注释和粘贴位置说明
  - 行业模板推荐（模板名称、字段说明、策略说明）
- [ ] 报告中的代码块保持英文（代码本身就是英文）
- [ ] 报告中的 YouTube 视频链接保持英文（不翻译标题）

#### 4. 模块文案双语化
- [ ] MOD-001：延迟检测结果的说明文案、CDN 优化建议、加速前后对比预估文案
- [ ] MOD-002：安全评分卡文案、表单结构分析文案、优化建议文案、4 个行业模板（模板名称、字段标签、策略说明）
- [ ] MOD-003：内容完整性评分项名称、博客矩阵指南、白皮书指导文案、缺失清单文案
- [ ] MOD-004：追踪覆盖 checklist 文案、部署代码注释、验证步骤文案

#### 5. 翻译文件管理
- [ ] 翻译 key 命名规范：`模块名.组件名.文本标识`，如 `mod02.securityScore.recaptchaTitle`
- [ ] 新增翻译文本时，必须同时提供中英文两个版本（不允许只加中文）
- [ ] 未找到翻译时降级为英文（fallback: en-US），不显示 raw key
- [ ] 提供一个翻译缺失检测脚本（开发环境运行），扫描代码中是否有硬编码中文未提取

#### 6. SEO 与路由
- [ ] URL 不携带语言参数（如 `?lang=en`），保持简洁
- [ ] HTML `lang` 属性随当前语言切换：`<html lang="zh-CN">` 或 `<html lang="en-US">`
- [ ] 页面 title 和 meta description 支持双语

### 接口契约

```
GET /api/user/language
  Headers: Authorization: Bearer {accessToken}
  Response: { language: "zh-CN" | "en-US" }

PATCH /api/user/language
  Headers: Authorization: Bearer {accessToken}
  Body: { language: "zh-CN" | "en-US" }
  Response: { success: true, language }
```

### 优先级
p1_high（v1.4 第二个迭代首位，可与 FEAT-301/302 部分并行）

### 预估工作量
L（3-5 天）—— 涉及前端全量文本提取 + 后端报告模板双语化 + 模块文案翻译

### 依赖
- v1.2 全量前端页面
- v1.2 报告引擎（PDF / Markdown / HTML 生成逻辑）
- v1.2 四个模块的诊断结果文案

### 外部依赖
无

### 备注
- 建议开发线优先使用成熟的 i18n 库（如 react-i18next 或 react-intl），减少自研成本
- 翻译文件初始版本可由我（产品线）提供，开发线负责接入框架
- 模块内长文案（如行业模板策略说明）翻译量较大，可分两期：第一期先翻译 UI 和报告，第二期翻译模块内详细文案

## v1.4 迭代排期建议

### 迭代 1（2 周）
- FEAT-301 诊断历史记录页（前端交互补齐 + 后端 API + 持久化存储）
